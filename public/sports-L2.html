1 心智模型

1.1 體育系統的本質——不是「下注平台」，是「多場館賽事聚合入口」

這個區別是 PM 做體育第一件要記住的事。體育系統的目的不是讓玩家下注，下注只是其中一條路徑。玩家打開體育，實際上有三條可能動線：

動線目的PM 要找誰
下注選賽事 → 進詳情 → 加購物車 → 送單體育 PM（自己）
觀賽看主播解說 / 比賽直播streamer 模組 PM（不是直播 PM！見 §4）
跟單 / 爆料看其他玩家的發文、複製對方的注單爆料模組 PM（gossiping）
PM 寫體育需求時的第一個動作：搞清楚你要動的是三條動線的哪一條。因為上版節奏可能不同、Owner 可能不同、相依的其他模組完全不同。

1.2 四層核心抽象（背下來）

體育系統的資料結構是四層嵌套 + 一個特殊結構：

場館
→
賽事 (event)
→
盤口 (market)
→
選項 (selection)
+ 串關 parlay
跨賽事的盤口組合，獨立結算邏輯

場館共 8 個：fb / im / ob / saba / yoo / im-esport / liveCasino / ws168。

PM 腦袋裡每次思考體育需求都會用到這四層。新 PM 第一天就要背下來，其他所有細節都是長在這上面。
對 RD 講話時：描述需求時一定要指明你改的是哪一層。「我要改賽事顯示」RD 會反問「是賽事層的時間/比分、還是盤口層的賠率？」混著講會被打回票。

1.3 PM 最常錯的三個概念（這一節最值錢）

① 不是每個盤口都有「球頭 K」

這是新 PM 最常掉的坑：

盤口類型有 K 嗎範例
大小盤 (Over/Under)✅ 有2.75、3.5
讓球盤 (Handicap)✅ 有+0.5、-1.25
勝負、首進球、波膽…❌ 沒有—
為什麼這個致命：PM 寫 UI 需求時如果預設「所有盤口都顯示 K 值」，在勝負盤上會顯示空白或亂碼。你的需求規格要明確標註「哪些盤口顯示 K」，不能假設。

② 賠率是場館給的，我們不能改

業主最常問：「能不能調整賠率讓玩家下注？」

答案：不行。

理由——我們是聚合器，不是做市商。賠率從場館即時下發，技術上沒有管道修改（程式裡 oddsvalue 永遠 read-only，前端只做 calcOdds 在不同賠率制之間換算馬/港/歐/印四種顯示）。

所有「動賠率」的需求要立刻辨識出來並當場拒絕，不是轉給 RD 評估。或者轉成「動返水 / 送紅利 / 改優惠 / 改流水要求」的變形需求——這些是可控的。
這是體育系統最根本的權力邊界。對業主，這個邊界不能讓步。

③ 場館不對稱

8 個場館（fb / im / ob / saba / yoo / im-esport / liveCasino / ws168）不是對等的：

  • needSportTokenVendorList = [fb, ob, saba, im-esport] — 只有這四個要拿 sport token
  • reserveBetVendor = [fb] — 預約下注只有 fb 支援
  • 只有 saba / fb / ob 有獨立的 simpleMarketConfig，其他場館 fallback 到 default
  • 簡易盤規則、串關規則、提前結算能力各場館不同
對 PM 的意義：任何「全體育生效」的需求，先問 RD：im / yoo / liveCasino / ws168 有沒有包含在內？ 看到「預約投注」類需求立刻警覺——只有 fb 有。

1.4 「看起來小、實際很大」的警報——購物車

體育系統所有路徑最後都匯到一個地方：投注購物車（stores/carts.ts，1172 行）。購物車看起來只是個確認頁，但它是：

  • 單注 vs 串關邏輯的匯合點
  • K 值顯示與否的判斷點
  • 賠率即時更新的處理點
  • 送單前所有驗證的閘門（最低金額、最高賠率、場館維護、風控、VIP、綁定、流水）
  • 五種下注來源的收斂點：simpleMarket / pushOrder / follow / chase / firstPay
結論：任何「只是改一下購物車」的需求——預設為大需求，工時 +30%。業主跟你說「很快啦、小改動」的時候，立刻警覺、回去對 RD 確認範圍再回覆。

1.5 已知的重大結構性事實（從程式碼反推）

這幾件事不寫在任何規格書裡，但你必須知道：

① pages/match-detail 不是「體育賽事詳情頁」，是共用殼

這個目錄被 93 個檔案反向引用，包括：open-room（開房對賭）、deposit（充值直播房）、gossiping（爆料）、live-casino（真人娛樂）、chat（聊天室）、floatingPopupManager（新手彈窗）。重構 match-detail = 重構半個 App，任何「把體育詳情頁獨立出來」的提議都要先做依賴盤點。

② pages/schedule/ 和 pages/result/ 是 copy 版

  • pages/schedule/index.vue 實質是 pages/live/components/Schedule 的 copy
  • pages/result/index.vue 實質是 pages/mine/match-result 的 copy

維護這兩個頁面的時候要想到源頭也要動（或反過來：改源頭時要看 sports 的 copy 有沒有要同步）。

③ 任務系統的條件引擎被體育列表頁引進來了

pages/home/index/sport/components/MatchList.vue 裡 import 了 @luxon/missions 的 MISSION_CONDITION_LV1_MAP，透過 useFindPageMission 整合。改任務條件類型會影響體育列表顯示——這跟任務系統 L2 §2.1 的「共用引擎地雷」對得上。任務系統的 PM 改條件，要回測體育列表頁。

  • 分頁順序與內容（聊天 / 競猜 / 賽況 / 包賠 / 爆料 / 下注單）
  • 底部 tab 的 key 和 component 映射
  • 賠率制計算公式（馬 / 港 / 歐 / 印 換算）
  • 購物車所有彈窗的文案與顯示邏輯
  • 場館差異化（哪個場館要 token、哪個支援預約）
  • 聊天室的跳轉引導路徑（/mine/rewardsTasks、/deposit、/home/sport、/bind-phone）
  • UI 色系 / 皮膚變數
  • 動畫（串關入籃、PK、進球）
  • 賠率類型 enum 定義
  • 分頁類型（PERIOD_TYPES）
PM 的直覺規則：「改一下 tab 順序」「把這個按鈕挪位置」「改這段文案」——九成是要發版。不要跟業主說「應該可以即時生效」。

6.1 四大紅線（Hanniber 定調，不可妥協）

#反模式為什麼不行PM 應對
1自己改賠率oddsvalue 從場館下發，前端只能 calcOdds 做顯示換算。沒有任何加價/減價/偏移機制直接拒絕，不要轉 RD 評估

6.2 其他疑似反模式（🚧 待主管補完）

下列是從程式碼推出來、需要主管確認的反模式：

  • 🚧 自己做串關保險 / 保本方案（現有靠 cash-out 或 combo 擺弄，額外方案需確認）
  • 🚧 自動下注 / 追漲追跌（自動化交易屬重大合規風險）
  • 🚧 後台直接改使用者已下注的訂單（訂單不可逆原則）
  • 🚧 前端截取直播畫面做二次分發（版權風險）
  • 🚧 跨場館合併盤口（把 fb 的上半場跟 ob 的下半場湊一張單，技術做不到）
  • 🚧 鎖盤期間偷偷接收下注（islocked=true 就該擋，不能繞過）
  • 🚧 把預約下注功能（只有 fb）做到其他場館（場館不支援）

請主管在下次 review 時確認這幾項是否應升格為正式紅線。

▶ L2-sport-001 新版賽事分頁套用任務條件引擎後派獎大爆炸 TSD-5200 共用引擎 high
發生經過

NBA 全明星賽活動 3（活動 ID 106，aId 10020）於 2026-02-17 由業主反饋異常：活動詳情頁任務完成進度全部顯示為 0，同時有多個會員在未達條件的狀況下獲得獎勵。案例會員 qianchao5566 單筆最大虧損 1825 卻跳到第二檔（888→2000），w151846 單筆最大虧損 588 卻領到第一檔 888，甚至有會員 tieu6581 根本沒投注指定賽事 ID 卻領到第一檔。

踩坑細節

新版賽事分頁的活動（aId 10020）沿用 conditionTypeLv1 三層條件引擎，跟任務系統 Mission（30000 系列）共用同一批後端邏輯。但新版賽事分頁是 GOR-5659 在 2025-09 才上的，做條件引擎對接時只驗了常見條件，沒充分測「單筆-負盈利金額」這種比較極值的條件類型，導致後端把「累加總和」當成「單筆最大」。

避雷 Checklist
  • 新的活動類型（aId）接任務條件引擎時，列出所有 conditionTypeLv1 類型並逐一跑 smoke test
  • 「單筆-極值型」條件必須單獨寫 test case
  • 派獎前先驗進度條的數字（「進度全顯示 0」就是警報）
  • 新 aId 第一次上線建議開監控型活動（單一業主、小流量）
  • L2 讀任務系統檔 §2.1 確認條件引擎共用清單
PM 事前能怎麼預見

接到「新版賽事分頁加條件類型」這種需求時（GOR-6228 / 6368 / 6339），PM 應該強制要求後端把所有既有條件類型跑一遍回歸，而不是只測新加的。共用引擎類改動，壞的永遠是你沒想到要測的那個。

▶ L2-sport-002 新版賽事分頁註冊限制沒等滿 24 小時就解鎖 TSD-5189 資料遷移 medium
根因

新版賽事分頁接上 registerRestrictions 時，沒把任務系統 GOR-6463 的教訓（改註冊限制要重跑註冊工具）同步套用，業主打開新活動但後端沒重新註冊會員資料，結果活動一開當天就讓不到 24 小時的新會員看到第二天的活動。

避雷 Checklist
  • 改 registerRestrictions 必須在需求單明寫「需排註冊工具重跑」
  • 新 aId 接註冊限制欄位前，先讀任務系統 L2 §5.2
  • QA 驗收要包含「24 小時前 vs 後新會員」兩種帳號
▶ L2-sport-003 新版賽事分頁的「展示時間」跟「活動時間」雙軌定義沒對齊 TSD-5042 共用引擎 low
根因

賽事分頁前端照「展示時間」排序/顯示，但運營活動後台送的卻是「活動時間」欄位，兩套時間的語義沒對齊。

避雷 Checklist
  • 新版賽事分頁的需求規格必須畫時間軸圖（展示開始 / 活動開始 / 活動結束 / 展示結束）
  • 跟運營對齊時用同一張圖確認語義
▶ L2-sport-004 新版賽事自動化活動壓垮 MongoDB 釀 S 級事故 TSD-5266 技術瓶頸 high
根因

新版賽事分頁的「自動化活動」在重置任務設定與進度時，對 MongoDB 發出大量寫入，導致 DB 忙碌，開自動化的活動全部無法正常開啟。

避雷 Checklist
  • 新活動類型引入「自動化 / 定時重置」必須標 [技術瓶頸] 風險
  • 上線前找 SRE 估容量、跑壓測
  • 第一個月請 Josephine 監控 DB 負載
PM 事前能怎麼預見

preventable=false。純技術瓶頸，PM 不用自責，但知道一下判估工時有感——「自動化 / 批次」類需求要主動加 +20% buffer。

▶ L2-sport-005 直播競猜開獎失敗 prod/test schema drift TSD-5305 版本切換 high
根因

後端表格 schema 變更只上到 test 沒同步到 prod，正式環境出現未預期的查詢結果，直播競猜盤口封盤與設賽果都失敗。

避雷 Checklist
  • schema 變更需求必須打 [schema 變更] 風險標記
  • 上版前跟後端三方確認 prod / test / slave 對齊
  • 排在流量低谷時段
▶ L2-sport-006 體育注單表 mysql/tidb 切換要先拆 task GOR-6205 技術瓶頸 medium
根因

cloud_data_game_transaction_sport 原本掛在 task_cloud_no_behind_inc 共用任務下面同步，要切換到 TiDB 查詢必須先獨立拆 task，而且切換過程中還要搭配 percona-toolkit 跑 alter table。

避雷 Checklist
  • 「mysql/tidb 切換」「house keeping」「DDL」需求預設 +30% 工時
  • 找 SRE 排上人力
  • QA 步驟通常 8 步起跳
▶ L2-sport-007 熊貓體育 TG 機器人被砍全業主炸 28 分鐘 TSD-5303 跨模組耦合 high
根因

熊貓體育場館的 prod 服務重啟依賴一個外部 TG（Telegram）機器人，該機器人被管理員或第三方砍掉後，prod 服務重啟會失效，導致熊貓體育頁面無法正常顯示。

避雷 Checklist
  • 新場館上線前要求後端寫依賴清單（外部機器人 / IP 白名單 / webhook / 推送服務）
  • 三方斷線預設 +20% 工時
▶ L2-sport-008 APP 盤口消失反覆復現 H5 沒事 TSD-5326 版本切換 high
根因

APP 端跟 H5 端的盤口取得 / 更新邏輯不一致（APP 是 Uni-APP，H5 是 Vue-H5），某個舊 bug 在 H5 修掉了 APP 沒跟上。

避雷 Checklist
  • 體育 bug 修復單強制拆 [前端][Vue-H5] + [前端][Uni-APP] 雙子單
  • 只修 H5 沒修 APP 的子單必須掛 reminder
  • APP 下版節奏要考慮審核期
▶ L2-sport-009 下注全場卻變下半場（購物車 race condition） TSD-5151 跨模組耦合 high
根因

會員在購物車送出下注時，前端傳給後端的 marketId / selectionId 跟下注當下卡片顯示的不一致，疑似盤口切換動畫或購物車 state 管理的 race condition 導致選錯盤口。

避雷 Checklist
  • 盤口切換動畫 / 重整時必須清空購物車 state
  • 易混淆玩法（全場/上半場/下半場）顯示文字加完整前綴
  • QA 必測「快速切換盤口後立刻送下注」的 race condition
▶ L2-sport-010 APP 投注顯示讓分 -5.5 實際變 -6.5 TSD-5073 版本切換 high
根因

會員下注時盤口顯示讓分 -5.5，FB 體育的盤口在下注送出與結算之間由三方改成了 -6.5，系統接受了 -6.5 的新盤口下注但 APP 前端沒即時同步顯示。

避雷 Checklist
  • 盤口在下注送出到成交之間變動，規則必須明確：拒絕 / 接受新盤口 / 彈窗確認
  • 若「接受新盤口」，前端必須阻塞下注流程顯示新賠率
  • 下注結果頁顯示「下注時盤口 vs 成交盤口」差異
▶ L2-sport-011 APP 串關下注金額快捷鍵異常 全業主 TSD-5085 版本切換 high
根因

APP 串關購物車的金額快捷鍵（+100/+500/+1000）沒做上限檢查，多次點擊累加時沒比對場館限額，直接加到超過限額就被自動設為場館限額最高金額。H5 同時有另一個 bug：快捷鍵不受會員餘額限制。

避雷 Checklist
  • 快捷鍵邏輯必須是 shared util，APP 和 H5 不能各做一份
  • 下注金額測四組邊界（min / max / 場館限額 / 會員餘額）
▶ L2-sport-012 代理後台串關注單只顯示第一場 TSD-5099 跨模組耦合 medium
根因

代理後台、雲平台、遊戲前台三套介面各自有一份串關注單的查詢/顯示邏輯，前台和雲平台改成了「整張串關單的輸贏及狀態」，但代理後台沒同步。

避雷 Checklist
  • 串關注單的顯示規則三端（前台 / 雲平台 / 代理後台）都要對齊
  • 需求單必須列出三端 owner
▶ L2-sport-013 在線人數拉高直播 + 體育投注一起卡 TSD-5145 技術瓶頸 high
根因

在線人數瞬間拉高使部分機器負載接近滿載，直播列表、聊天室、體育下注的盤口載入全部受影響——這幾個功能在 infra 層共用同一批 API gateway 或 DB 連線池。

避雷 Checklist
  • 大賽事時段必須避開（杯賽開幕、世界盃、NBA 季後賽）
  • 新功能上線前找運維預先擴容
  • 「直播 + 體育」跨模組需求預設 +20% 工時
