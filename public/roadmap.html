我們現在的問題

PM 改 A 功能的時候，常常撞到 B 功能——因為對 B 不熟、沒看到耦合、踩了別人踩過的坑。

這個現象不是個人能力問題，是系統性問題：產品越複雜、功能越耦合，PM 靠個人記憶撐不住。而且新 PM 上來的學習曲線越來越陡——一年前的 PM 可以「靠經驗」，現在的 PM 經驗不夠就會一直翻車。

THE GOAL

我們最終想要的

讓系統可以自動產制需求——業主一句話，系統輸出完整的需求規格。PM 從「寫規格的人」變成「審核與戰略判斷的人」。

但從現狀到這個目標中間有一段很大的落差。這份文件把落差具體化，並說明我們現在做的 L1/L2/L3 知識庫在這個大圖裡的位置。

1

核心洞察：這兩件事不是同一軸

很多人會直覺以為「L2 做得夠完整 → 自然就能自動產需求」。這是錯的假設。

L1 / L2 / L3 知識庫

輸入：系統（程式碼、Jira）
輸出：給人讀的文件
目的：讓人讀懂系統
挑戰：抽象 + 分類
失敗：文件太多沒人看

自動產需求系統

輸入：業主的意圖（一句話）
輸出：給 RD 執行的規格
目的：讓系統讀懂人
挑戰：消除歧義 + 影響範圍
失敗：需求寫錯誤導實作
KEY TAKEAWAY

方向相反的兩件事

你不會因為寫了很多 L2，系統就「自然」會產需求。L2 會成為自動產需求系統的燃料，但燃料不等於引擎。兩件事要分開推進、但最終會合流。

2

解剖 PM 寫需求這件事

要談「自動化」就要先解剖「手動」到底做了什麼。PM 寫需求實際上是一個翻譯 + 校對的過程：

業主說：
「我想讓新手玩家第一次存款後看到包賠彈窗」
↓
PM 腦袋裡跑的事（現在靠經驗）：
  1. 翻譯：「新手」是誰？綁手機算嗎？首存算嗎？
  2. 翻譯：「第一次存款」什麼時機觸發？
  3. 翻譯：「包賠彈窗」是現有的還是新的？
  4. 校對：這會動到哪些現有流程？
  5. 校對：會不會跟已有的彈窗衝突？
  6. 校對：各場館表現一致嗎？
  7. 校對：有沒有類似需求踩過坑？
  8. 輸出：寫成規格、畫時序、列驗收
↓
規格文件 → RD 實作
落差的具體位置

L1/L2/L3 只解決 4-7，沒解決 1-3 和 8

我們現在做的知識庫解決的是「校對」階段——影響範圍、跨模組、踩坑、場館差異。但「翻譯」階段（1-3）和「結構化輸出」階段（8）完全沒碰。這就是你感覺到的落差的具體位置。

3

從現狀到終極目標的五個階段

這不是一步跳過去的。中間有五個現實的里程碑，每個階段都有明確的能力提升、清楚的下一步。

STAGE 0

PM 從零手寫

靠個人經驗加搜尋。產出品質跟 PM 資歷成正比。新 PM 翻車頻率高、資深 PM 是 bottleneck。

STAGE 1 現在

PM 手寫 + 知識庫輔助查詢

PM 寫之前先查 L1 / L2 / L3，少踩坑、少漏影響範圍。這就是我們現在在做的事——L1 瀏覽、L2 經驗、L3 API 三層知識庫。

這階段的成效指標

  • 新 PM onboarding 時間縮短
  • 同一個坑不再重複踩
  • 跨模組影響漏算頻率下降
STAGE 2 下一步

PM 寫草稿 + Agent 審稿

PM 寫完需求初稿 → 丟給 agent 審 → agent 對照 L2 pitfalls + 模組邊界，回報「這個需求漏了 A、B、C 點，類似需求踩過 TSD-XXXX 的坑」。

為什麼這是最現實的下個跳點

  • L1 / L2 / L3 已經備好了（燃料齊全）
  • 「審稿」比「寫稿」技術門檻低太多——只需要檢查完整性，不需要解決歧義
  • 產出是「已有稿 + 缺漏清單」，不是從無到有
  • PM 仍然負責最終決定——不是被取代，是被檢查
STAGE 3

業主提需求 + Agent 起草 + PM 校稿

業主用一句話丟需求 → agent 查 L2 + 歷史類似需求 → 起草 5-7 成完整度的草稿 → PM 審一遍改錯、補戰略判斷。

這階段的關鍵依賴

  • Agent 需要「類似需求的歷史樣本」做參考
  • 需要結構化的「業主訴求 → 最終規格 → 實作結果」配對資料
  • 這個資料庫是我們目前完全沒有的東西——Jira 有單但沒有結構化的回饋迴圈
STAGE 4

Agent 起草 + 自動影響範圍檢查 + PM 核准

同階段 3，但 agent 會主動跑影響範圍檢查（跨模組依賴圖、歷史踩坑配對、版本排程衝突）。PM 只負責核准或退回。

STAGE 5

Agent 全自動 + 例外才找 PM

Agent 自信度夠高、歷史類似需求成功率夠高時，直接出規格給 RD。PM 只處理 agent 標「不確定」的需求，專注在戰略判斷與例外處理。

4

我們從 Stage 1 到 Stage 5 之間欠的三件事

從現在的階段到終極目標，落差不是「知識庫深度不夠」，而是三個我們還沒開始建的東西：

落差 A

系統輸入端還是「人話」

業主的一句話要變成結構化的需求意圖——這需要 意圖識別 和 歧義消除對話能力。業主說「新手」，agent 要會反問「你的新手定義是 A 還是 B？」。

這塊 L2 完全幫不上忙，是獨立的能力建構。

落差 B

沒有「需求 → 結果」的結構化歷史

我們知道 Jira 有哪些單、知道踩過哪些坑，但不知道：過去類似需求最後做出來的規格長什麼樣、拆成幾張子單、上線後改了幾次。

這是自動產需求最重要的燃料，但 Jira 單沒有「業主原始訴求 → 最終實作規格」的配對資料。現在不開始收集，一年後也還是沒有。

落差 C

沒有「需求完整性」的可驗證標準

PM 寫完需求怎麼知道夠完整？現在靠 review 會議、靠經驗。自動化需要可量化的完整性判斷——覆蓋所有受影響模組？列出跟歷史類似需求的差異？標明版本排程與回滾策略？

這個「完整性 schema」我們也沒有，而且需要由資深 PM 定義才有意義。

5

所以我們該做什麼

不要跳級追 Stage 5，要老實地為 Stage 2 鋪路。

短期（接下來半年到一年）

1. 繼續把 L2 做完（7 個模組）
  1. 這是燃料，是 Stage 1 → Stage 2 必備的前置。現在只做完體育，還有活動、直播、存提款、娛樂城、我的、系統、其他。
2. 建立「需求審稿 agent」試點
  1. 這就是 Stage 2，離現狀最近的跳點。
  2. 用現有 L2 + pitfalls 讓 agent 能審 PM 的需求草稿、回報缺漏。
  3. 成功了就推給全 PM 使用，失敗了也能知道 L2 深度還不夠，回頭補。
3. 開始收集「需求 → 結果」的結構化歷史
  1. 每次 PM 寫完需求、RD 做完、QA 測完、上線後，把「業主原始訴求 / PM 初稿 / 最終規格 / 實際實作 / 是否踩坑」五欄填成表。
  2. 這是 Stage 3 的燃料。現在不收集，一年後也還是沒有。
  3. 格式可以從簡單開始——一個 Google Sheet 或 Notion table 都行，重點是養成記錄習慣。

中期（一年後）

試 Stage 3：Agent 起草 + PM 校稿
  1. 有了審稿 agent + 需求歷史資料庫之後，讓 agent 嘗試從零起草需求。
  2. 目標是「5-7 成完整度」，剩下的 PM 補戰略判斷。

長期

Stage 3 穩定後再談 Stage 4 / 5
  1. 自動影響範圍檢查、例外處理、全自動出規格。
  2. 這階段的挑戰已經不是 L2 夠不夠深，而是 信心度模型 和 例外偵測——那是另一個量級的工程。
ONE-LINER

「讓 PM 理解需求成本下降」到「系統自動產需求」之間，差的不是知識庫深度，而是三個我們還沒開始建的東西：

  1. 意圖識別（業主 → 結構化需求）
  2. 需求-結果配對的歷史資料庫
  3. 需求完整性的可驗證 schema

L2 / L3 會成為這些的燃料，但不是引擎本身。

要一步一步來，中間有個很現實的跳點叫 審稿 agent，那是我們做完 L2 之後該瞄準的下個目標。
