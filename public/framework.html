0為什麼要分層

早期我們只有「把功能寫清楚」一種目標，結果寫出來的文件總是掉進三個陷阱之一：

太淺

PM 寫需求還是翻車

規格寫完了，上線一樣爆，因為跨模組影響沒寫進去

太深

淪為技術文件

塞滿 API / DB / 檔案路徑，PM 看了用不上、RD 有程式碼

太雜

沒人想維護

功能改一次文件就全部過期，整理的人心累、看的人不信

根本原因：用同一種顆粒度、同時服務所有讀者。

分層的核心洞察是：不同資訊有不同的保鮮期、不同的讀者、不同的使用時機。把它們混在一起整理，就是在用維護成本最高的方式、服務最窄的族群。

1四層定義

最短版

L1
WHAT
現在這功能長什麼樣
L2
WHY IT BREAKS
動這功能會怎麼爆炸
★ CP 值最高
L3
HOW IT INTERFACES
這功能怎麼接外面
L4
HOW IT'S IMPLEMENTED
這功能內部怎麼寫

完整對照

層回答的問題主要讀者使用時機保鮮期
L1這功能是什麼？PM / QA / 業主寫需求、驗收、跟業主對齊隨版本變（短）
L2這功能有哪些坑？中階以上 PM寫需求前的盤點、估影響範圍長壽（越用越值）
L3這功能怎麼運作？PM ↔ RD 溝通介面估工、跟 RD 對話、debug 討論跟程式改（中）
L4這功能怎麼實作？RD / 架構師寫程式、改 bug跟程式改（短）

2L1 vs L2 的邊界（最容易混淆）

很多人以為「把規格寫細一點就是 L2」——錯。

  • L1 寫得再細仍然在描述「功能本身」（what）
  • L2 的本質是「歷史 + 關聯」（why it breaks）

L2 的三個判斷特徵

  • 可追溯到具體事件：Jira 單、上線事故、運營抱怨
  • 跨功能：不是單一功能內部邏輯，是跟別人的耦合
  • 反直覺：直覺能想到的就不算 L2（是 L1）

任務系統對照舉例

L1任務系統有簽到、每日、新手三種
直覺可得、功能本身
L2三種並存不能砍舊的，因為舊 aId 還被運營活動關聯（GOR-6437）
反直覺、跨模組、有事故證據
L1新手任務完成後獎勵進背包
功能規則
L2改任務條件會動到轉盤/裂變/賽事，因為共用條件引擎（GOR-6359）
結構性耦合、踩過坑

  • 會改 → L4（實作細節、PM 別碰）
  • 不會改 → L3（對外介面、PM 要懂）

4每層「夠用就好」的門檻

層不足的症狀過度的症狀（越界）
L1需求業主看不懂、QA 測不出來把 UI 細節列 100 頁（Figma 就有）
L2需求上線爆炸、同個坑踩兩次PM 寫起技術分析報告
L3跟 RD 對話雞同鴨講、估工完全沒底PM 開始畫 ER 圖、設計 API
L4—（PM 不碰）PM 去改 .vue 檔

5PM 能力校正標準

資歷要求層級驗收方式
初階 PM獨立完成 L1，L2 要 review需求文件有「跨模組影響」章節
中階 PM獨立完成 L1 + L2寫得出「歷史坑 + 避雷計畫」
資深 PML1 + L2 + L3 輔助判斷能對 RD 估工提出質疑

關鍵原則：L1 + L2 是門檻、L3 加分、L4 不碰。

6過期管理策略

關鍵洞察：L1 / L3 會一直過期、L2 大多不會過期。花力氣整理 L2，報酬率最高。

  • L1 接受過期：每個版本動到的部分，該版 PM 順手更新。不要幻想一次整理完。
  • L2 只增不減：踩坑只會累積不會消失。新坑寫進去，舊坑就算功能重做也還是歷史教訓。
  • L3 讓它過期：需要時對著程式碼重生成，不值得手動維護。
  • L4 不整理：程式碼本身就是 L4。

什麼會改，什麼不會改

類型範例改變頻率
當下狀態現在任務有 16 種條件、aId=30000每版都可能變
歷史事件GOR-6373 redis 頂不住永遠不會變（已發生）
結構性關係條件引擎是跨活動共用的很少變（改了就是大重構）
決策與教訓加條件類型要前後端同版越用越鞏固

7整理知識庫的四個真正目的

避免「為了整理而整理」，請時時校準這四個目的：

① 把個人經驗變成組織資產

現在的現實是——功能的坑只在資深 PM 腦袋裡。那個 PM 離職、調組、請產假，知識就歸零。新 PM 接手就是再踩一次同樣的坑。整理 L2 = 把腦袋裡的東西變成檔案裡的東西。功能會改，但「為什麼會翻車」的教訓會留下來。

② 降低判斷成本，不是消除判斷

知識庫真正的用途是：讓 PM 做判斷時不用每次從零開始想。沒知識庫，PM 想「加這個條件會不會有問題？」要翻 Jira 兩小時還可能漏。有知識庫，翻踩坑總表 30 秒——「喔這是共用引擎類，要列測試範圍」。功能改了，具體條件可能不一樣，但「這類改動屬於共用引擎類」的判斷框架不會變。

③ 建立團隊溝通的共同語言

當你跟同事說「任務系統要注意跨模組」，她點頭——但她心裡的「跨模組」跟你心裡的「跨模組」可能不是同一件事。有了 L2 文件，你可以說「去翻 3-activities-mission-L2.md §2 依賴地圖」——大家看到的是同一份東西。這個價值跟功能改不改無關，是對齊的價值。

④ 讓知識庫本身成為能力評估工具

L1 / L2 / L3 分層就是一把尺：新 PM 來，給她 L1 文件看她能不能寫出需求、給 L2 文件看她能不能列出影響範圍、寫需求時漏了 L2 寫過的坑——這是能力問題不是資訊問題。沒有文件時你無法區分「她不知道」和「她不會判斷」。有文件後，責任歸屬變清楚。

∞一句話總結

整理知識庫的目的不是記錄「功能現在長什麼樣」（那會過期），

而是沉澱「這個功能怎麼思考、怎麼避雷、怎麼跟別人溝通」（這會累積）。

功能會變，思考框架不會。
你在整理的其實是後者。
